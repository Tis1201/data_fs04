# WebRTC Signaling & ICE Candidate Flow (Python/aiortc)

## Overview
WebRTC enables real-time peer-to-peer communication (video, audio, data) between browsers or native clients. However, **the actual connection setup is complex**, involving multiple asynchronous steps and message exchanges over a separate signaling channel (e.g., WebSocket).

This guide explains the signaling and ICE candidate exchange flow used in this project, with reference to the Python `aiortc` implementation and the test clients in this repo.

---

## 1. Signaling Channel
- WebRTC peers cannot connect directly on their own—they need a signaling server (here, via WebSocket) to exchange information required to set up the connection.
- The signaling server is only used for coordination, not for media/data transfer.

## 2. Connection Setup Flow

### a. Peer Connection Initialization
- Each peer creates an `RTCPeerConnection` object.
- Event handlers are registered for:
    - `icecandidate` (for local ICE candidates)
    - `connectionstatechange` (for connection state)
    - `datachannel` (for incoming data channels)

### b. Offer/Answer Exchange
- One peer (the offerer) creates an SDP offer and sends it to the other via the signaling server.
- The answerer receives the offer, creates an SDP answer, and sends it back.
- Both peers set the received SDP as their remote description.

### c. ICE Candidate Gathering & Exchange
- As the peer connection is set up, each peer discovers network routes (ICE candidates) it can use to connect.
- Each time a new local ICE candidate is found, the `icecandidate` event fires.

---

## 3. Which Messages Go Over Which Channel?

### WebSocket (Signaling) Channel
All signaling/coordination messages go over WebSocket. No real data or media is sent here.

| Direction        | Message Type      | Description                                                                                      |
|------------------|------------------|--------------------------------------------------------------------------------------------------|
| Offerer → Signaling Server → Answerer | `offer`           | Contains the SDP offer generated by the offerer’s RTCPeerConnection.                             |
| Answerer → Signaling Server → Offerer | `answer`          | Contains the SDP answer generated by the answerer’s RTCPeerConnection.                           |
| Either → Signaling Server → Other     | `ice-candidate`   | Contains a single ICE candidate discovered by either peer. Sent as soon as each is discovered.   |

**Example message:**
```json
{
  "type": "webrtc",
  "data": {
    "type": "ice-candidate",
    "candidate": "candidate:...",
    "sdpMid": "0",
    "sdpMLineIndex": 0,
    "timestamp": "...",
    "_clientMessageId": "..."
  }
}
```

### WebRTC Peer-to-Peer Channel
All real-time data and media flows directly between peers after the connection is established.

| Direction | Message Type          | Description                                  |
|-----------|----------------------|----------------------------------------------|
| Peer ↔ Peer | Data channel messages | Any application data (chat, files, etc).      |
| Peer ↔ Peer | Media streams         | Audio/video streams (if used).                |

**Example message:**
```json
{"text": "Hello from peer!"}
```

---

### Summary Table

| Step                | Channel   | Who Sends      | Message Type      | Purpose                        |
|---------------------|-----------|---------------|-------------------|--------------------------------|
| Offer SDP           | WebSocket | Offerer       | offer             | Start connection setup         |
| Answer SDP          | WebSocket | Answerer      | answer            | Complete connection setup      |
| ICE Candidate       | WebSocket | Both          | ice-candidate     | Discover network paths         |
| Data Channel Open   | WebRTC    | Both          | (any)             | App-level data (chat, etc)     |
| Media Stream        | WebRTC    | Both          | (media)           | Audio/video                    |

---

**Key Points:**
- All signaling (offer, answer, ICE candidates) is over WebSocket.
- All real data (chat, files, media) is over the WebRTC peer-to-peer connection.
- The signaling server never sees your real data.

- The peer sends this candidate over the signaling channel to the other peer.
- When a peer receives a remote ICE candidate, it calls `addIceCandidate()` to add it to its connection.
- This process continues until both sides have enough candidates to establish a viable connection.

### d. Data Channel
- Once the connection is established, the data channel is opened and ready for use.

## 3. Message Structure & Deduplication

### ICE Candidate Message Example
```
{
  "type": "webrtc",
  "data": {
    "type": "ice-candidate",
    "candidate": "candidate:...",
    "sdpMid": "0",
    "sdpMLineIndex": 0,
    "timestamp": "2025-04-18T12:54:43.123Z",
    "_clientMessageId": "ice-candidate-2025-04-18T12:54:43.123Z-<uuid>"
  }
}
```
- Each candidate message includes a unique `_clientMessageId` and `timestamp`.
- When sending, the message ID is stored locally.
- When receiving, if the message ID matches one sent by this peer, it is ignored (prevents self-processing).

---

## 4. Why This Is Hard
- WebRTC is inherently asynchronous and stateful—many things happen in parallel and order is not always guaranteed.
- ICE negotiation can be fast or slow depending on network conditions.
- Both peers must implement the same signaling protocol and handle edge cases (duplicates, race conditions, NAT, etc).
- Debugging requires careful logging and understanding of the full flow.

---

## 5. Key Implementation Points
- **Always send local ICE candidates as soon as they are discovered.**
- **Always add remote ICE candidates as soon as they are received.**
- **Never process your own ICE candidates (deduplicate using message IDs).**
- **Log all actions** for debugging and traceability.

## 6. How to Set Up Video Streaming

### 1. Add a Video Track to the Peer Connection

**Python (aiortc) Example:**
```python
from aiortc import RTCPeerConnection, VideoStreamTrack

class CustomVideoStreamTrack(VideoStreamTrack):
    async def recv(self):
        # Generate or grab a video frame here
        ...

pc = RTCPeerConnection()
video_track = CustomVideoStreamTrack()
pc.addTrack(video_track)
```

**Browser (JavaScript) Example:**
```js
const pc = new RTCPeerConnection();
navigator.mediaDevices.getUserMedia({ video: true, audio: false })
  .then(stream => {
    stream.getTracks().forEach(track => pc.addTrack(track, stream));
  });
```

### 2. Signal as Usual
- The offer/answer SDP will automatically include video media sections if a video track is present.
- ICE candidate exchange and signaling are unchanged.

### 3. Receiving Video
- **Python:** Listen for the `"track"` event and process/display video frames.
- **Browser:** Attach the incoming stream to a video element.

### 4. No Special Signaling Messages for Video
- You do **not** need to send special WebSocket messages for video.
- The presence of a video track is automatically negotiated via SDP in the offer/answer.

### 5. Video Setup Summary Table

| Step              | Code/Action                           | Channel    | Notes                       |
|-------------------|---------------------------------------|------------|-----------------------------|
| Add video track   | `pc.addTrack(video_track)`            | WebRTC     | Before offer/answer         |
| Signal SDP        | Send offer/answer as usual            | WebSocket  | SDP includes video section  |
| ICE candidates    | Send/receive as usual                 | WebSocket  |                             |
| Receive video     | Handle `track` event                  | WebRTC     | Frames delivered to peer    |

## 8. Transitioning to Peer-ID Based Architecture (TBD)

### Overview
For IoT device remoting and streaming applications, a peer-ID based architecture offers significant advantages over room-based connections. This section outlines the planned transition to a peer-ID model.

### Why Peer-ID for IoT Devices?

| Feature | Room-Based | Peer-ID Based |
|---------|------------|---------------|
| Addressing | Indirect (via room) | Direct (via device ID) |
| Connection Model | Many-to-many | One-to-one |
| Device Management | Requires room tracking | Natural mapping to physical devices |
| Server Complexity | Higher (room state) | Lower (simple registry) |
| Scaling | More complex | More straightforward |

### Implementation Plan

1. **Device Registration**
   - Each IoT device registers with a unique peer ID (e.g., serial number)
   - Server maintains a registry of online devices

2. **Connection Process**
   ```
   ┌─────────────┐                ┌─────────────┐                ┌─────────────┐
   │             │                │             │                │             │
   │  IoT Device │◄──Register──►  │  Signaling  │◄──Register──►  │   Browser   │
   │  (peer-123) │                │   Server    │                │  (peer-456) │
   │             │                │             │                │             │
   └─────────────┘                └─────────────┘                └─────────────┘
          ▲                                                             │
          │                                                             │
          │                         Connection                          │
          │                         Request to                          │
          │                         peer-123                            │
          │                                                             │
          └─────────────────────────────────────────────────────────────┘
                               Direct WebRTC
                             Data & Video Stream
   ```

3. **Message Structure**
   ```json
   // Device Registration
   {
     "type": "register",
     "peerId": "device-123",
     "capabilities": ["terminal", "video"]
   }

   // Connection Request
   {
     "type": "connect",
     "targetPeerId": "device-123"
   }
   ```

4. **Benefits for Remote Terminal & Streaming**
   - Simpler connection management
   - Clear mapping between physical devices and WebRTC peers
   - Easier authentication and access control
   - Reduced server-side state management
   - Better scaling for many devices

### Compatibility Plan
- Maintain backward compatibility with room-based connections
- Implement peer-ID as an alternative connection method
- Gradually transition to peer-ID as the primary method for IoT devices

## Recent Changes (2025-04-19)

- **Security:** Only room participants can send or receive signaling messages (offer, answer, ICE). Unauthorized attempts are logged and rejected.
- **Participant Management:** Users are now explicitly added as participants when joining a room, ensuring up-to-date and accurate room state.
- **Centralized Permission Checks:** All signaling handlers now verify participant status before processing messages.
- **Architecture:** The codebase is moving toward a composition-based pattern, with a planned `RoomWebRTCController` for modular and secure signaling logic. Room handles membership; the controller will handle signaling and always reference the Room for checks.
- **Testing:** All changes have been validated with the Python `aiortc` client and custom test scripts. Logs confirm correct enforcement.
- **Impact:** Test clients (and real clients) must join the room before sending any signaling messages. If not, their messages will be rejected and a warning will appear in the server logs.

This improves security, prevents leaks, and sets the stage for further modularization and maintainability.
